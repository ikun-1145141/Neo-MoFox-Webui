## 1. 概述与目标

### 1.1 功能目标

为 Neo-MoFox-WebUI 提供完整的插件管理能力，支持以下核心功能：

| 功能分类 | 具体能力 |
|---------|---------|
| **插件生命周期管理** | 查看已加载插件列表、查看插件详细信息、重载插件 |
| **组件查询** | 查看插件包含的所有组件（Action、Adapter、Command、Router、Agent 等） |
| **配置管理集成** | 判断插件是否有配置、支持从插件详情跳转到配置页面 |
| **状态监控** | 展示插件加载状态、组件运行状态 |

### 1.2 设计原则

1. **上层 API 优先**: 所有业务逻辑只调用 `src/app/plugin_system/api/` 下的扁平化 API，不直接操作 Core 层
2. **类型安全**: 所有数据模型使用 Pydantic 定义，前端 TypeScript 类型与后端严格对应
3. **统一响应**: 所有接口返回统一的 `BaseResponse<T>` 结构
4. **职责分离**: Router 层仅负责接口暴露，业务逻辑由 Manager 层处理
5. **只读优先**: 当前版本仅支持插件重载，不支持插件加载/卸载（敏感操作）

## 3. API 端点设计

### 3.1 路由基础信息

| 属性 | 值 |
|-----|-----|
| **路由名称** | `plugin` |
| **基础路径** | `/api/plugin` |
| **CORS 策略** | `["*"]`（开发阶段）|
| **认证要求** | 所有端点需要 `VerifiedDep` |

### 3.2 端点列表

#### 3.2.1 获取插件列表

**端点**: `GET /api/plugin/list`

**功能描述**: 获取所有已加载插件的摘要信息列表

**请求参数**: 无

**响应结构**:
```json
{
  "code": 200,
  "message": "success",
  "data": [
    {
      "plugin_name": "webui",
      "plugin_description": "Neo-MoFox WebUI 插件",
      "plugin_version": "1.0.0",
      "is_loaded": true,
      "has_config": true,
      "component_count": 8,
      "component_types": ["router", "event_handler", "service"]
    }
  ]
}
```

**核心逻辑**:
1. 调用 `plugin_api.list_loaded_plugins()` 获取所有已加载插件名称
2. 对每个插件调用 `plugin_api.get_manifest()` 获取基础信息
3. 调用 `config_api.get_config(plugin_name)` 判断是否有配置
4. 通过 `registry.get_by_plugin(plugin_name)` 统计组件数量和类型

---

#### 3.2.2 获取插件详情

**端点**: `GET /api/plugin/{plugin_name}`

**功能描述**: 获取指定插件的详细信息，包括完整组件列表

**路径参数**:
- `plugin_name` (string): 插件名称

**响应结构**:
```json
{
  "code": 200,
  "message": "success",
  "data": {
    "plugin_name": "webui",
    "plugin_description": "Neo-MoFox WebUI 插件",
    "plugin_version": "1.0.0",
    "is_loaded": true,
    "has_config": true,
    "plugin_path": "/path/to/plugin",
    "manifest": {
      "name": "webui",
      "version": "1.0.0",
      "author": "MoFox Studio",
      "license": "GPL-3.0"
    },
    "components": [
      {
        "signature": "webui:router:dashboard",
        "component_type": "router",
        "component_name": "dashboard",
        "description": "仪表板路由",
        "status": "active",
        "extra": {
          "custom_route_path": "/api/dashboard"
        }
      }
    ],
    "dependencies": []
  }
}
```

**核心逻辑**:
1. 调用 `plugin_api.get_plugin(plugin_name)` 校验插件是否存在
2. 调用 `plugin_api.get_manifest(plugin_name)` 获取清单
3. 调用 `registry.get_by_plugin(plugin_name)` 获取所有组件
4. 对每个组件提取类型、名称、状态等信息
5. 针对特定组件类型提取扩展属性（如 Router 的 `custom_route_path`）

**错误场景**:
- 插件不存在: `{"code": 404, "message": "插件未找到"}`
- 插件未加载: `{"code": 400, "message": "插件未加载"}`

---

#### 3.2.3 重载插件

**端点**: `POST /api/plugin/{plugin_name}/reload`

**功能描述**: 重载指定插件（卸载后重新加载）

**路径参数**:
- `plugin_name` (string): 插件名称

**请求体**: 无

**响应结构**:
```json
{
  "code": 200,
  "message": "插件重载成功",
  "data": {
    "success": true,
    "plugin_name": "webui",
    "reload_time": "2026-05-04T12:00:00Z"
  }
}
```

**核心逻辑**:
1. 调用 `plugin_api.reload_plugin(plugin_name)`
2. 返回重载结果和时间戳

**错误场景**:
- 插件不存在: `{"code": 404, "message": "插件未找到"}`
- 重载失败: `{"code": 500, "message": "插件重载失败: {错误原因}"}`

**权限要求**: 需要管理员权限

---

#### 3.2.4 获取插件组件列表

**端点**: `GET /api/plugin/{plugin_name}/components`

**功能描述**: 获取指定插件的所有组件，支持按类型筛选

**路径参数**:
- `plugin_name` (string): 插件名称

**查询参数**:
- `component_type` (string, optional): 组件类型筛选（如 `router`, `action`, `adapter`）

**响应结构**:
```json
{
  "code": 200,
  "message": "success",
  "data": [
    {
      "signature": "webui:router:dashboard",
      "component_type": "router",
      "component_name": "dashboard",
      "description": "仪表板路由",
      "status": "active",
      "extra": {
        "custom_route_path": "/api/dashboard",
        "cors_origins": ["*"]
      }
    },
    {
      "signature": "webui:adapter:web_adapter",
      "component_type": "adapter",
      "component_name": "web_adapter",
      "description": "Web 适配器",
      "status": "active",
      "extra": {
        "platform": "web"
      }
    }
  ]
}
```

**核心逻辑**:
1. 如果指定了 `component_type`，调用 `registry.get_by_plugin_and_type(plugin_name, component_type)`
2. 否则调用 `registry.get_by_plugin(plugin_name)` 获取所有组件
3. 对每个组件提取详细信息，包括：
   - 基础信息：签名、类型、名称
   - 运行状态：对于 Adapter/Router 等有状态组件，查询实际运行状态
   - 扩展属性：根据组件类型提取特有属性

**组件类型对应的扩展属性**:

| 组件类型 | 扩展属性 |
|---------|---------|
| **Adapter** | `platform` (平台名称) |
| **Router** | `custom_route_path` (路由路径), `cors_origins` (CORS 配置) |
| **Command** | `permission_level` (权限等级), `command_name` (命令名) |
| **Action** | `primary_action` (是否为主动作), `action_name` (动作名) |
| **Agent** | `agent_name` (代理名), `usables` (可用工具列表) |

## 5. 业务逻辑设计

### 5.1 Manager 层职责

**文件位置**: `Plugin/managers/plugin_manager.py`

**核心职责**:
1. 调用 Neo-MoFox 插件 API 获取原始数据
2. 数据转换：将 Core 层对象转换为 WebUI 数据模型
3. 数据聚合：组合多个 API 调用结果
4. 错误处理：统一捕获和转换异常

### 5.2 关键方法设计

#### 5.2.1 list_plugins()

**功能**: 获取插件列表

**调用流程**:
```
1. plugin_api.list_loaded_plugins() 
   → 获取插件名称列表

2. 对每个插件并行执行：
   a. plugin_api.get_manifest(name)
      → 获取版本、描述等基础信息
   b. config_api.get_config(name)
      → 判断是否有配置（返回值非 None）
   c. registry.get_by_plugin(name)
      → 获取组件字典，统计数量和类型

3. 转换为 PluginSummary 列表
```

**异常处理**:
- 单个插件信息获取失败时，记录日志但不影响整体列表
- 所有插件都失败时，返回空列表并记录错误

---

#### 5.2.2 get_plugin_detail()

**功能**: 获取插件详情

**调用流程**:
```
1. plugin_api.is_plugin_loaded(name)
   → 校验插件是否已加载（未加载则抛出异常）

2. plugin_api.get_plugin(name)
   → 获取插件实例

3. plugin_api.get_manifest(name)
   → 获取完整清单

4. registry.get_by_plugin(name)
   → 获取所有组件

5. 对每个组件调用 _extract_component_info()
   → 提取组件详细信息

6. manifest["dependencies"]["components"]
   → 提取依赖列表

7. 聚合为 PluginDetail 对象
```

**组件信息提取逻辑** (`_extract_component_info()`):
```
1. 从签名解析：plugin_name, component_type, component_name

2. 读取组件类的 __doc__ 作为描述

3. 根据 component_type 判断状态：
   - Router: 调用 router_api.get_mounted_router(signature)
     → 已挂载 = active，否则 = inactive
   - Adapter: 调用 adapter_api.is_adapter_active(signature)
     → 已启动 = active，否则 = inactive
   - 其他: 默认 active

4. 提取扩展属性：
   - Router: custom_route_path, cors_origins
   - Adapter: platform
   - Command: permission_level, command_name
   - Action: primary_action, action_name
   - Agent: agent_name, usables

5. 返回 PluginComponentInfo
```

---

#### 5.2.3 reload_plugin()

**功能**: 重载插件

**调用流程**:
```
1. plugin_api.is_plugin_loaded(name)
   → 校验插件存在

2. await plugin_api.reload_plugin(name)
   → 执行重载

3. 记录重载时间（UTC）

4. 返回 PluginReloadResult
```

**异常处理**:
- 重载失败时捕获异常，返回 `success=false` 和错误信息
- 特殊处理循环依赖错误、配置错误等

---

#### 5.2.4 get_plugin_components()

**功能**: 获取插件组件列表（支持类型筛选）

**调用流程**:
```
1. 如果指定 component_type：
   registry.get_by_plugin_and_type(name, ComponentType[type])
   
   否则：
   registry.get_by_plugin(name)

2. 对每个组件调用 _extract_component_info()

3. 返回 list[PluginComponentInfo]
```

## 7. 关键约束与注意事项

### 7.1 API 调用约束

| 约束项 | 说明 |
|--------|------|
| **只使用上层 API** | 所有业务逻辑只调用 `src/app/plugin_system/api/` 下的函数，不直接导入 Core 层 |
| **异步操作** | 插件的加载/卸载/重载都是异步函数，必须使用 `await` |
| **签名格式** | 组件签名统一为 `plugin_name:component_type:component_name` |
| **注册表查询** | 使用 `get_global_registry()` 获取全局注册表单例 |

### 7.2 组件状态判断

| 组件类型 | 状态判断依据 |
|---------|-------------|
| **Router** | 调用 `router_api.get_mounted_router(signature)` 是否返回实例 |
| **Adapter** | 调用 `adapter_api.is_adapter_active(signature)` |
| **Action/Agent** | 每次聊天动态创建，无持久状态，始终为 `active` |
| **Service** | 单例模式，插件加载后始终 `active` |
| **Command** | 注册后常驻，始终 `active` |
| **EventHandler** | 注册后常驻，始终 `active` |

### 7.3 权限控制

**端点权限要求**:

| 端点 | 权限 | 说明 |
|-----|------|------|
| `GET /list` | ✅ 需要认证 | 所有已登录用户可访问 |
| `GET /{plugin_name}` | ✅ 需要认证 | 所有已登录用户可访问 |
| `POST /{plugin_name}/reload` | ⚠️ 需要管理员权限 | 仅管理员可重载插件 |
| `GET /{plugin_name}/components` | ✅ 需要认证 | 所有已登录用户可访问 |

**权限实现**:
- 基础认证: 在路由上添加 `dependencies=[VerifiedDep]`
- 管理员权限: 在端点函数内调用权限检查 API（待实现）

### 7.4 错误处理规范

**统一错误响应**:
```json
{
  "code": 500,
  "message": "具体错误描述",
  "data": null
}
```

**错误码约定**:

| HTTP 状态码 | 业务 code | 场景 |
|-----------|----------|------|
| 200 | 200 | 成功 |
| 404 | 404 | 插件未找到 |
| 400 | 400 | 插件未加载 / 参数错误 |
| 500 | 500 | 内部错误 / 插件重载失败 |
| 403 | 403 | 权限不足 |

**异常捕获策略**:
```python
try:
    result = await manager.some_method()
    return BaseResponse.ok(data=result)
except PluginNotFoundError as e:
    raise HTTPException(status_code=404, detail=str(e))
except PermissionError as e:
    raise HTTPException(status_code=403, detail=str(e))
except Exception as e:
    logger.error(f"操作失败: {e}", exc_info=True)
    raise HTTPException(status_code=500, detail=f"操作失败: {str(e)}")
```

### 7.5 性能优化建议

1. **并行查询**: 在 `list_plugins()` 中，对每个插件的信息获取可以并行执行（使用 `asyncio.gather()`）
2. **缓存机制**: 对于不常变化的清单信息，可以在 Manager 层添加短期缓存（60秒）
3. **分页支持**: 当插件数量较多时（>50），考虑添加分页参数（`page`, `page_size`）
4. **延迟加载**: 组件详细信息在详情页才获取，列表页只展示摘要

### 7.6 日志记录规范

**关键操作日志**:
- 插件重载: `logger.info(f"插件 {plugin_name} 重载成功")`
- 错误捕获: `logger.error(f"获取插件信息失败: {e}", exc_info=True)`
- 组件状态异常: `logger.warning(f"组件 {signature} 状态异常")`

**日志级别**:
- `DEBUG`: 详细调用流程
- `INFO`: 关键操作成功
- `WARNING`: 非致命错误（如单个插件信息获取失败）
- `ERROR`: 严重错误（如整体查询失败）
